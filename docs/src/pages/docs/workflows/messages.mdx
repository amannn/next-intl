# Linting messages

To ensure quality and completeness of your messages, you can lint them with the [eloqnt CLI](https://studio.eloqnt.dev).

**Usage:**

```bash
npm install -D @eloqnt/cli
npx eloqnt lint
```

**Output:**

```
NewsArticle.title
✖ Missing translation in de.json (missing-translation)
```

Based on static analysis of your messages and source code, `eloqnt lint` can detect issues like:

1. **Missing translations**: A message in the source locale has no translation in a target locale
2. **Inconsistent arguments**: A translation's ICU arguments don't match the source message
3. **Superfluous keys**: A target locale defines a key that the source locale doesn't
4. **Orphan messages**: A catalog message is never referenced in your source code
5. **Undefined keys**: Your source code references a key that no catalog defines

The CLI reads its configuration from `.eloqnt/config.ts`, closely mirroring what you might already know from [`createNextIntlPlugin`](/docs/usage/plugin):

```tsx filename=".eloqnt/config.ts"
import {defineConfig} from '@eloqnt/cli';

export default defineConfig({
  srcPath: './src',
  messages: {
    path: './messages',
    locales: 'infer',
    sourceLocale: 'en',
    format: 'json'
  }
});
```

Both `.json` and `.po` catalogs are supported, including those generated via [`useExtracted`](/docs/usage/extraction).

<Callout>

Since `eloqnt lint` fails with a non-zero exit code when errors are found, it can be integrated into your continuous integration workflow—see the [eloqnt docs](https://studio.eloqnt.dev/docs) for details.

</Callout>
