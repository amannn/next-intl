# AI agents

AI agents like Claude Code and Cursor are increasingly writing code in our apps—including user-facing labels. This page covers how you can set up your project so that agents produce well-internationalized code, and how translations fit into an agentic workflow.

## Usage with `useExtracted` [#use-extracted]

[`useExtracted`](/docs/usage/extraction) was purpose-built for agents as well as humans: Messages are written inline right where they're used, while `next-intl` takes care of extracting them to your message catalogs.

```tsx
import {useExtracted} from 'next-intl';

function InlineMessages() {
  const t = useExtracted();
  return <h1>{t('Look ma, no keys!')}</h1>;
}
```

This approach has some relevant advantages for agents:

- **No naming of keys**: Agents don't need to come up with key names, avoiding inconsistent naming conventions.
- **Local reasoning**: Adding, updating, or removing messages only requires editing a single file—no need to read lengthy message catalogs into the context window.
- **Always in sync**: Catalogs are updated automatically, therefore agents can't forget to add a message or leave dead ones behind.
- **Refactoring-friendly**: Moving code across components doesn't require restructuring namespaces or keys.

Learn more about the motivation for this API in the introduction blog post: [useExtracted: The Tailwind of i18n?](/blog/use-extracted)

## Usage with `useTranslations` [#use-translations]

If you're using [`useTranslations`](/docs/usage/translations), it can be helpful to add instructions to an agents file like `CLAUDE.md` or `AGENTS.md` to avoid hardcoded strings ending up in your app:

```markdown filename="AGENTS.md"
## Internationalization

- All user-facing strings are rendered with `useTranslations`
  from `next-intl`, never hardcoded in components.
- Messages are added to `messages/en.json`, other locales are
  handled by translators.
- Namespaces mirror component names (e.g. `UserProfile`).
- ...
```

Make sure to adapt these instructions to the conventions of your project. General recommendations that apply to agents and humans alike:

1. **Use descriptive, yet short key names**: Keys like `title` or `submit` within a component namespace describe the intent of a message without repeating the message itself (see [structuring messages](/docs/usage/translations#structuring-messages)).
2. **Use ICU arguments instead of string concatenation**: Composing sentences from multiple messages doesn't survive translation to languages with a different word order (see [ICU messages](/docs/usage/translations#icu-messages)).
3. **Lint your messages**: Agents can forget to update catalogs after refactoring. A [linter for your messages](/docs/workflows/messages) can catch issues like missing translations and unused messages, and provides agents a feedback loop to correct themselves.

## Translating messages [#translating-messages]

While agents can help with many aspects of internationalization, translating message catalogs is a task where they typically don't perform particularly well:

1. **Missing context**: A catalog entry alone doesn't tell where and how a message is used in your app, oftentimes leading to translations that miss the intent.
2. **Consistency**: Terminology tends to drift across a catalog and between sessions, especially as your app grows.
3. **Language nuances**: Aspects like plural rules, grammatical gender, and formality levels in target languages require careful consideration that generic agents often gloss over.
4. **Context pollution**: Editing catalogs for many locales fills up the context window with content that's mostly irrelevant for the agent's main task.

Due to this, translation is usually better handled by a dedicated workflow. [eloqnt/studio](https://studio.eloqnt.dev) provides a translation pipeline that was designed for `next-intl`: It analyzes your source code to enrich messages with usage context, applies per-locale styleguides, and verifies translations after they've been generated—all from the command line.
